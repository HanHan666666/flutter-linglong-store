# 应用详情页头部操作区 UI 优化设计

## 背景

当前应用详情页头部在多轮迭代后出现明显视觉拥挤：

- 应用图标、标题、描述、标签、版本、仓库、主按钮、桌面快捷方式、卸载、分享、安装状态消息都集中在同一个 `Row + Wrap` 内。
- 主按钮、次级按钮、危险按钮、分享按钮视觉权重接近，用户难以快速判断当前最重要动作。
- 安装状态消息贴在按钮下方，像后补信息，失败态、复制入口和普通进度态缺少统一容器。
- 页面已有业务规则是正确的，但视觉编排没有把“主操作 / 次操作 / 状态反馈”分层。

本次优化聚焦应用详情页头部的操作区域，目标是让打开、安装、更新、创建桌面快捷方式、卸载和分享这一组交互更美观、大方、专业，同时不改变现有安装、更新、打开、卸载和快捷方式创建链路。

## 现状确认

### 页面与组件位置

- 详情页主体：`lib/presentation/pages/app_detail/app_detail_page.dart`
- 主按钮组件：`lib/presentation/widgets/install_button.dart`
- 详情页次级操作：`lib/presentation/widgets/app_detail_secondary_actions.dart`
- 详情页信息区：`lib/presentation/widgets/app_detail_info_section.dart`
- 安装飞行动画：`lib/presentation/widgets/install_to_download_flyout.dart`

### 当前业务规则

- 主按钮三态继续保持：
  - 未安装：安装
  - 已安装且可更新：更新
  - 已安装且无更新：打开
- 安装中 / 排队中继续由 `installQueueProvider` 的任务状态驱动。
- “创建桌面快捷方式”和“卸载”只在 `installedAppsProvider` 确认当前 `appId` 存在本地安装实例时展示。
- 卸载必须继续走 `AppUninstallFlow.run()` 和 `appUninstallServiceProvider`，不得新增页面内卸载流程。
- 创建桌面快捷方式继续调用 `LinglongCliRepository.createDesktopShortcut(appId)`，不得新增零散 shell 调用。
- 安装 / 更新成功入队后继续触发 `InstallToDownloadFlyoutLayer`，动画源点仍使用详情页头部应用图标。

## 目标

1. 重塑详情页头部操作区，让用户一眼看出主操作和次级操作。
2. 让“安装 / 更新 / 打开”按钮在详情页拥有更稳定、更精致的尺寸、色彩和状态表达。
3. 让“创建桌面快捷方式 / 卸载 / 分享”形成有秩序的次级操作组。
4. 将安装状态消息独立为状态条，避免挤压按钮和标题区域。
5. 保持现有业务链路、状态判断、通知机制、无障碍语义和安装飞行动画行为不变。
6. 在桌面宽度变化时保持稳定布局，不出现按钮随机换行、文本溢出或状态消息压缩主按钮的问题。

## 非目标

- 不重做整个详情页为完整双栏页面。
- 不调整截图、应用介绍、应用信息、评论区、版本列表的数据流。
- 不新增“更多”菜单，也不把桌面快捷方式或卸载重新折叠进菜单。
- 不修改安装队列、卸载服务、快捷方式创建仓储的业务实现。
- 不新增后端接口，不新增 ll-cli 调用。

## 方案对比

### 方案 A：保守微调

在现有 `Wrap` 内调整按钮间距、颜色和尺寸。

优点：

- 改动最小。
- 对现有测试影响较少。

缺点：

- 仍然是一个大 `Wrap` 承载所有动作，视觉层级只能局部改善。
- 主按钮、次级按钮和状态消息仍然缺少清晰结构。
- 难以达到“精致、专业”的观感。

### 方案 B：头部操作区结构化重做（采用）

抽出详情页头部组件，按“应用身份信息 + 主操作 + 次级操作 + 状态条”重新组织。

优点：

- 改动聚焦在最混乱的区域，范围可控。
- 不影响详情页下半部分数据流。
- 显著改善视觉层级和响应式稳定性。
- 符合 `docs/03d-ui-pages.md` 中“次级操作与主按钮同一行展示”的业务约定。

缺点：

- 需要补充头部布局测试，避免后续迭代重新退化。
- 需要为详情页按钮增加专用 `hero` 尺寸。

### 方案 C：完整双栏详情页

参考 `docs/03e-aurora-layer-linux-desktop-design-system.md`，将整个详情页改为“左主右辅”。

优点：

- 页面整体现代化程度最高。
- 能同时优化版本历史、评分摘要、开发者信息等区域。

缺点：

- 改动范围大，会影响截图、评论、版本列表等多个区域。
- 更容易引入滚动、响应式和测试回归。
- 超出本次用户明确提到的打开、安装和桌面快捷方式区域。

## 采用方案

采用 **方案 B：头部操作区结构化重做**。

本次只处理详情页头部和头部操作区，让“主操作”和“次级操作”明确分层。页面下方区域保持现有结构，代码调整仅限头部组件拆分带来的调用迁移。

## 详细设计

### 1. 头部结构

新增一个详情页头部组件，命名为 `AppDetailHeroHeader`。

组件职责：

- 展示应用图标、应用名、描述、标签、版本和仓库等身份信息。
- 根据外部传入状态渲染主操作按钮。
- 渲染已安装态下的桌面快捷方式和卸载入口。
- 渲染始终可见的分享入口。
- 渲染安装 / 更新任务状态消息。

组件不负责：

- 判断安装状态。
- 调用安装、更新、打开、卸载、快捷方式创建等业务逻辑。
- 直接读取 Riverpod Provider。

设计原因：

- 保持页面层负责业务聚合，头部组件负责纯渲染。
- 避免把详情页状态判断复制进子组件，符合当前项目“页面聚合后下发轻量 props”的性能要求。
- 完整双栏改版时继续复用这个头部组件。

### 2. 桌面宽屏布局

当可用宽度充足时，头部采用三段结构：

1. 左侧应用图标：`96 x 96`，圆角 `16`，继续使用 `_installSourceKey` 包裹，保证安装飞行动画源点不变。
2. 中间应用信息：应用名、简短描述、标签、版本 / 仓库 / 架构辅助信息。
3. 右侧操作面板：主按钮占据第一行，次级操作组在第二行。

视觉规则：

- 头部容器使用轻面板样式：白底 / 深色主题表面色、细边框、`16px` 圆角、无阴影。
- 应用名保持详情页主标题层级，不降低为小字号。
- 描述最多两行，标签使用轻胶囊，不与按钮抢视觉。
- 右侧操作面板设置稳定最小宽度，避免主按钮随文字长度跳动。

### 3. 中窄宽度布局

当宽度不足以稳定展示三段结构时：

- 图标和应用信息仍在上方横向展示。
- 操作组换到信息区下方，保持主按钮优先，次级操作随后。
- 次级操作允许换行，但必须作为一个组换行，不让单个按钮随机插入应用文案区域。

当宽度更窄时：

- 主按钮占据更完整的一行。
- 桌面快捷方式、卸载、分享组成下一行。
- 文本按钮如果空间不足，优先通过 `Flexible`、最小宽度和 `TextOverflow.ellipsis` 保证不溢出。

### 4. 详情页主按钮

继续复用 `InstallButton` 的业务状态和点击行为，但为详情页提供更专业的视觉形态。

实现规则：

- 给 `InstallButton` 增加 `ButtonSize.hero`。
- 详情页主按钮视觉高度为 `48px`，宽度约束为 `minWidth: 144px`、`maxWidth: 184px`。
- 未安装和更新态使用实心主色按钮。
- 打开态使用浅底描边按钮，强化“已安装后的启动动作”而不是继续使用主安装视觉。
- 安装中继续展示进度，但在详情页尺寸下保证百分比、速度和取消图标不拥挤。
- 排队中保持等待态，hover 时显示取消安装的规则不变。

不改变：

- `InstallButtonState` 枚举语义。
- 安装、更新、打开、取消的回调。
- 下载速度来源和展示条件。

### 5. 次级操作组

重做 `AppDetailSecondaryActions` 的视觉，不改变可见性和回调。

规则：

- 已安装时展示：
  - 创建桌面快捷方式
  - 卸载
- 未安装时完全隐藏，不展示禁用态。
- 分享入口始终可见，使用轻量图标按钮展示。

视觉规则：

- “创建桌面快捷方式”使用轻量次级描边按钮，图标使用桌面快捷方式语义图标。
- “卸载”使用危险轮廓按钮，默认克制，hover / focus / pressed 时增强错误色背景。
- “分享”使用图标按钮，交互热区不低于 `48 x 48`。
- 次级操作高度与详情页主按钮视觉对齐，但主按钮通过颜色和宽度保持更高层级。

### 6. 状态消息条

安装任务存在 `displayMessage` 时，不再作为普通文字贴在按钮下方，而是渲染为头部内的状态条。

普通状态：

- 浅色信息条。
- 左侧显示状态图标。
- 中间显示完整来源文案的可视省略文本。
- Tooltip 和复制按钮继续使用完整文本，遵守当前“复制文案完整性约定”。

失败状态：

- 浅红背景并带错误色边框。
- 错误文本允许换行，不压缩复制按钮。
- 复制按钮使用完整错误详情：优先 `errorDetail`，其次 `errorMessage`，再回退 `rawMessage / displayMessage`，保持现有逻辑。

设计原因：

- 状态条把“动作反馈”和“操作入口”分开。
- 失败信息不再挤压主按钮，用户能更清楚地看到失败原因和复制入口。

### 7. 无障碍

本次必须保持或增强现有无障碍能力：

- 装饰性图标使用 `ExcludeSemantics`。
- 主按钮继续通过 `InstallButton` 提供安装、更新、打开、卸载语义。
- 分享、快捷方式、卸载按钮必须同时具备 tooltip 和语义标签。
- 可交互热区不低于 `48 x 48`。
- 状态消息的 Tooltip、复制按钮和可见文本必须使用完整语义，不复制省略后的短文案。

### 8. 性能

本次优化不应增加额外 Provider 监听。

规则：

- `AppDetailPage` 继续在页面层读取 `appDetailProvider`、`installQueueProvider`、`installedAppsProvider`、`updateAppsProvider`。
- 头部组件只接收已经计算好的 props。
- 不在 `build` 中新增 IO、解析或 ll-cli 调用。
- 不引入动画循环；只保留必要的 hover、状态切换和已有安装飞行动画。

## 测试设计

### Widget 测试

新增或调整详情页测试，覆盖：

- 未安装态：
  - 展示主按钮“安装”。
  - 不展示“创建桌面快捷方式”和“卸载”。
  - 分享入口仍可见。
- 已安装态：
  - 展示主按钮“打开”或“更新”。
  - 展示“创建桌面快捷方式”和“卸载”。
  - 点击次级操作仍触发现有回调。
- 安装中：
  - 展示进度按钮。
  - 展示状态条。
  - 复制按钮使用完整状态文本。
- 失败态：
  - 错误文本不被按钮挤压。
  - 复制按钮仍复制完整错误详情。
- 响应式：
  - 在较窄宽度下不出现 Flutter overflow。

## 验收标准

1. 详情页头部视觉层级清晰：主按钮最突出，次级操作整齐且不抢焦点。
2. 未安装、已安装、可更新、安装中、排队中、失败态都能正确展示。
3. 快捷方式和卸载仍只在本地已安装实例存在时展示。
4. 安装、更新、打开、取消、创建快捷方式、卸载、分享行为不变。
5. 安装飞入下载中心动画仍以详情页应用图标为源点。
6. 状态消息 Tooltip、复制和错误详情完整性不回退。
7. 相关 Widget 测试通过，`flutter analyze` 无错误。

- 左侧保留头部、截图、介绍、评论。
- 右侧放评分摘要、版本历史摘要、当前版本和开发者信息。
- 这属于页面级重构，不纳入本次头部操作区优化范围。
