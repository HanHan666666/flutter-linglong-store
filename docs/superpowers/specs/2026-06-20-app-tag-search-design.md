# 应用标签搜索开发方案

## 目标

应用详情页标签支持点击。点击后进入现有搜索结果页，标题栏搜索框显示不可拆分的 Tag 胶囊，并查询包含该标签的应用；结果允许包含当前应用。

## 后端设计

继续复用 `POST /visit/getSearchAppList`，新增可选参数 `tagName`、`tagLan`。普通文本搜索继续使用 `name`；标签搜索只发送标签参数，按 `tagName + tagLan` 精确匹配，不新增接口。

Tag 从 `ll_app_affiliated_dtls.app_tag_list` JSON 中彻底独立为 `ll_app_tag` 表：

| 字段 | 用途 |
| --- | --- |
| `id` | 主键 |
| `app_id` | 应用标识 |
| `tag_name` | 标签名称 |
| `lan` | 标签语言，如 `zh_CN`、`en_US` |
| `sort_order` | 保留上游标签展示顺序 |
| `create_time` | 创建时间 |

约束与索引：

- 唯一约束：`app_id + lan + tag_name`。
- 标签搜索索引：`lan + tag_name + app_id`。
- 详情查询索引：`app_id + lan + sort_order`。

定时任务按应用调用统一的 `replaceTags(appId, tags)` 事务方法：先删除该应用全部旧标签，再清洗、去重并批量写入新标签。明确返回 `appTagList=[]` 时只删除旧标签；批次请求失败、超时、解析失败或响应遗漏请求的 `app_id` 时保留旧标签。单个应用写入失败不得影响其他应用。

详情接口改为按 `app_id + lan` 批量读取 `ll_app_tag`，避免 N+1 查询；搜索接口通过索引精确过滤。同步完成且数据发生变化后统一失效相关详情与搜索缓存。

迁移完成并验证稳定后，删除 `app_tag_list` 字段、Tag JSON TypeHandler 及 Tag JSON Redis 缓存，禁止长期双写，确保 `ll_app_tag` 是唯一数据源。

## Flutter 设计

标签点击统一进入 `/search_list?tag=<名称>&tagLan=<语言>`。搜索页、分页、应用卡片和错误处理继续复用现有逻辑。

标题栏搜索框增加互斥的标签模式：

- 显示不可拆分、不可编辑的 Tag 胶囊。
- 胶囊可通过关闭按钮或 Backspace 删除；删除后回到普通文本搜索并获得输入焦点。
- 标签模式不触发普通文本搜索候选。
- 前进、后退和页面重建均从路由恢复 Tag。
- 标签点击与删除入口满足无障碍语义、键盘操作和最小交互尺寸要求。

Flutter 需保留 `AppTagDTO.language` 到领域模型，标签点击时原样传递名称与语言，禁止根据界面文本猜测标签身份。

## 迁移顺序

1. 新建 `ll_app_tag`，从现有 JSON 数据回填并核对数量。
2. 后端定时任务改为写入新表，详情和搜索接口改为读取新表。
3. Flutter 接入标签路由、Tag 胶囊和标签搜索参数。
4. 完成接口、Mapper、事务同步、缓存键、Provider、路由和 Widget 测试。
5. 稳定验证后删除旧 JSON 字段、处理器和缓存代码，并同步更新两端 `AGENTS.md` 与项目文档。

## 验收标准

- 点击详情页标签后，标题栏显示对应 Tag 胶囊并自动加载同标签应用。
- 标签匹配同时校验名称和语言，分页结果稳定且允许包含当前应用。
- 明确空标签可清除旧数据；请求失败或响应遗漏不会误删旧标签。
- 普通文本搜索、搜索候选、详情页和返回导航行为无回归。
- 后端查询命中索引，无 JSON 扫描和 N+1 查询；Flutter 静态分析及相关测试全部通过。
