# 下载中心复制安装日志按钮设计

**日期:** 2026-06-06
**状态:** 已确认
**范围:** 下载管理弹窗中的安装日志复制入口

## 背景

下载中心当前把 `_TaskCard` 整条任务项绑定为复制入口：用户点击应用名称、状态区域或卡片空白处都会复制 `InstallTask.commandOutput`。复制成功后还会调用全局 `showAppNotification()`，在底部显示 SnackBar。

这个行为与预期不一致。下载项本身应只负责展示任务状态和承载打开、取消、重试、移除等明确操作；复制安装日志必须是一个显式按钮，避免用户误触，也避免把整条卡片伪装成复制入口。

## 目标

- 下载项整条卡片不再响应点击复制。
- 每个存在 `commandOutput` 的任务项展示明确的“复制日志”按钮。
- 点击“复制日志”按钮后复制该任务自己的完整 `commandOutput`。
- 复制成功反馈只在按钮内部呈现：按钮文字短暂变为“复制成功”，约 1.2 秒后恢复“复制”。
- 不再为下载中心日志复制触发全局底部通知。
- 继续使用 `InstallTask.id` 作为任务记录身份，保持日志跟随任务记录生命周期。

## 非目标

- 不改变安装队列、取消、重试、打开、删除等业务行为。
- 不新增 `ll-cli` 调用，也不改变 `commandOutput` 的采集和保存逻辑。
- 不把日志复制改成弹窗预览、日志文件导出或批量复制。
- 不调整下载中心整体尺寸和分区结构。

## 方案

采用“任务卡片操作区内新增复制日志按钮”的方案。

实现上保留 `_TaskCard` 作为下载项的统一渲染组件，但移除 `InkWell.onTap` 的复制绑定。复制入口改为 `_TaskCard` 内部的显式 `TextButton`，按钮只在 `task.commandOutput.trim().isNotEmpty` 时展示。

复制按钮使用局部 `State` 管理成功反馈：

- 点击时调用 `Clipboard.setData(ClipboardData(text: output))`。
- 复制完成后设置 `_isOutputCopied = true`。
- 启动 1200ms 计时器，到期后恢复 `_isOutputCopied = false`。
- Widget dispose 时取消计时器。

为了避免操作区挤压现有按钮，复制按钮放在现有打开、取消、重试、移除按钮的左侧，并使用紧凑 `TextButton` 样式。这样当前任务、排队任务和历史记录都能在不改变队列行为的前提下复用同一套复制入口。

## 可访问性与文案

- 按钮文本使用国际化文案：
  - 新增 `copyLog`: “复制日志”
  - 新增 `copySucceeded`: “复制成功”
- `Semantics` 与 `Tooltip` 使用当前按钮文本，保证屏幕阅读器能读出复制成功状态。
- 装饰性图标不参与本次改动；复制入口使用文字按钮，符合用户明确要求。

## 测试策略

更新 `test/widget/presentation/widgets/download_manager_dialog_test.dart`：

1. 构造带 `commandOutput` 的历史任务。
2. 打开下载管理弹窗后点击任务名称，验证不会调用 `Clipboard.setData`。
3. 点击“复制日志”按钮，验证剪贴板写入完整 `commandOutput`。
4. 验证按钮文字显示“复制成功”。
5. 推进 1200ms 后验证按钮恢复“复制”。
6. 验证不再显示“命令已复制到剪贴板”的全局通知。

## 验收标准

- 下载 item 本体点击不会复制日志。
- “复制日志”按钮是唯一的日志复制入口。
- 复制成功反馈仅体现为按钮文字短暂变为“复制成功”。
- 下载中心日志复制不再出现底部 SnackBar。
- 相关 widget 测试通过，`flutter analyze` 无新增问题。
