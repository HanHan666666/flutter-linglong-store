# UI 设计规范 — 第五章：Aurora Layer Linux 桌面商店设计系统

> 文档版本：1.0  
> 创建日期：2026-04-23  
> 适用范围：`flutter-linglong-store` 设计规范与 HTML 设计 Demo  
> 设计方向：`Aurora Layer`  
> 气质优先级：`通用桌面应用商店 > 开发者 / 系统工具感 > 社区生态 / 开放平台感`

---

## 1. 背景与目标

当前 Flutter 商店已具备完整的功能骨架，包括：

- 应用外壳：`lib/presentation/widgets/app_shell.dart`
- 标题栏：`lib/presentation/widgets/title_bar.dart`
- 侧边栏：`lib/presentation/widgets/sidebar.dart`
- 通用应用卡片：`lib/presentation/widgets/app_card.dart`
- 应用详情页：`lib/presentation/pages/app_detail/app_detail_page.dart`
- 推荐页：`lib/presentation/pages/recommend/recommend_page.dart`
- 更新页：`lib/presentation/pages/update_app/update_app_page.dart`
- 下载管理：`lib/presentation/widgets/download_manager_dialog.dart`
- 设置页：`lib/presentation/pages/setting/setting_page.dart`

但现阶段界面仍存在以下问题：

- 全站视觉语言不够统一，首页、详情页、下载管理、设置页的观感割裂。
- 标题栏、侧边栏、主内容区和弹层缺少明确的桌面层级秩序。
- 主题系统仅支持浅色 / 深色模式，缺少统一的主题色能力。
- 页面间信息密度与展示密度不一致，容易出现“首页偏展示、内页偏默认 Material”的断层。
- 业务组件已有统一状态逻辑，但视觉表达尚未沉淀为设计系统。

本规范目标不是“重做一套更花的 UI”，而是将现有 Linux 应用商店收敛为一套：

- 适配 Linux 桌面语境
- 能统一浅色 / 深色 / 主题色
- 与当前 Flutter 架构和功能组件对齐
- 可为后续正式 Flutter 重构提供直接设计依据

的完整设计系统。

## 2. 设计定位

### 2.1 设计名称

`Aurora Layer for Linux Desktop Store`

### 2.2 设计关键词

- 轻盈
- 干净
- 有层次但不过度玻璃
- 现代桌面感
- 信息清晰
- 交互明确
- 可换主题色

### 2.3 非目标

以下方向不属于本次规范目标：

- 不将商店设计成纯内容运营站或网页风商城。
- 不引入移动端 App 风格的大面积留白和强手势语义。
- 不以开发者 IDE 风格主导整个商店气质。
- 不改现有信息架构的主路径：推荐、全部、排行、更新、我的应用、下载管理、设置等一级入口继续保留。
- 不改现有统一业务约束：搜索入口唯一、卡片三态按钮规则、卸载拦截、更新队列、右键菜单等已有行为契约。

## 3. 总体设计原则

### 3.1 Linux 桌面原生感优先

所有布局、组件和动效都以“桌面应用”而非“网页页面”为参照：

- 标题栏必须像桌面窗口壳层，而不是网页导航条。
- 下载管理和通知反馈更像工作面板，而不是移动端弹层。
- 右键菜单、键盘焦点、悬停反馈、窗口层级感必须被纳入设计。

### 3.2 商店展示力与工具效率并重

- 首页和推荐页需要承担内容发现与品牌展示。
- 列表页、详情页、更新页、下载管理需要承担快速浏览与任务决策。
- 当展示性与效率冲突时，内页优先效率，首页适当承担视觉表达。

### 3.3 统一视觉骨架

全站统一使用以下视觉秩序：

- Shell 容器
- 页面区块标题
- 面板 / 卡片层级
- 统一按钮与状态表达
- 统一的间距、圆角、阴影和过渡节奏

### 3.4 主题色是系统能力

主题色不仅用于主按钮，还必须驱动：

- 导航激活态
- 输入焦点
- 标签和胶囊选中态
- 进度与任务主轨
- 评分强调
- 通知和轻提示中的品牌性信息

### 3.5 默认平衡密度

本规范以 `平衡型` 为默认密度：

- 不使用营销站式大留白
- 不压缩到系统设置页级别的高密度
- 保证首页展示力与内页效率之间的稳定折中

## 4. 信息架构与 Shell 布局

### 4.1 Shell 结构

Aurora Layer 的主壳层统一采用：

`标题栏 + 左侧主导航 + 中央主工作区 + 可选右侧辅助区`

对应到 Flutter 当前结构：

- Shell：`AppShell`
- 标题栏：`CustomTitleBar`
- 左侧主导航：`Sidebar`
- 主工作区：各主路由页面 + 二级页面覆盖层

### 4.2 标题栏规范

标题栏承担以下职责：

- 左侧：Logo / 产品名称
- 中央：全局搜索输入框
- 右侧：窗口控制按钮 + 少量全局操作入口

规则：

- 搜索仍是全局唯一主搜索入口，搜索结果页不新增第二个主搜索框。
- 标题栏高度保持桌面窗口稳定比例，避免网页 Header 感。
- 窗口按钮必须贴近右侧边缘，保留桌面交互热区。
- 标题栏背景使用轻层次表面色，不做强玻璃、不做大面积高饱和色。

### 4.3 左侧导航规范

侧边栏继续使用现有信息架构，但重新定义视觉规则。

导航结构：

- 顶部固定入口：推荐 / 全部应用 / 排行 / 更新
- 中部动态分类：办公 / 系统 / 开发 / 娱乐等
- 底部固定入口：我的应用 / 下载管理 / 设置

规则：

- 一级导航是全站稳定骨架，不承担内容推荐职责。
- 激活态通过主题色软底 + 图标强化 + 字重提升表现。
- Hover 与 active 分离：hover 只做轻反馈，active 才是明确选中。
- 折叠态保留图标与 Tooltip，展开态优先单行可读性。

### 4.4 中央主工作区规范

中央主工作区是所有内容页面的统一承载区：

- 页面顶层统一边距和区块间距
- 页面标题、二级标题、说明文案采用统一层级
- 列表类、详情类、任务类、设置类页面共享一套基础面板规则

### 4.5 右侧辅助区规范

右侧辅助区是可选区，不是固定广告栏。

可承载内容：

- 今日推荐 / 今日精选
- 推荐专题
- 详情页摘要信息
- 开发者其他应用
- 当前下载 / 更新概览

规则：

- 不是每个页面都显示
- 仅当存在“辅助决策信息”时出现
- 在中等及以下宽度时可自动收起
- 禁止成为长期固定的装饰性空白栏

## 5. 主题系统

### 5.1 主题模式

支持：

- 跟随系统
- 浅色模式
- 深色模式

### 5.2 主题色方案

官方预设色板建议为 `7` 套：

1. 默认蓝
2. 生态绿
3. 活力橙
4. 湖青
5. 暖玫
6. 紫靛
7. 石板灰蓝

额外支持：

- 跟随系统强调色

### 5.3 主题色职责

主题色用于以下语义层：

- `accent`：主激活色
- `accentSoft`：选中背景 / 轻提示底色
- `accentBorder`：焦点边框 / 选中边框
- `accentGlow`：轻高光、选中光晕、图表强调

### 5.4 功能状态色独立

以下语义色不得被主题色覆盖：

- 成功：`success`
- 警告：`warning`
- 危险：`danger`
- 禁用：`disabled`
- 错误：`error`

换主题色后，按钮和导航可以变化，但：

- “安装成功”
- “更新失败”
- “危险操作”
- “任务警告”

等功能语义必须保持直观和稳定。

## 6. 基础视觉令牌

### 6.1 字体

建议以 Linux 中文桌面稳定字体栈为主：

- `Noto Sans CJK SC`
- `Source Han Sans SC`
- `WenQuanYi Micro Hei`
- 系统 sans fallback

字号层级沿用当前文档体系并做 Aurora Layer 语义归整：

- Display：32
- Headline L：28
- Headline M：24
- Title：20
- Body：16
- Body M：14
- Caption：13
- Tiny：12

规则：

- `16px` 仍为主正文底线
- `12px` 仅用于标签 / 徽章 / 极度受限空间
- 列表页和详情页主要阅读内容禁止退化成 `12px`

### 6.2 间距

采用 `4 / 8 / 12 / 16 / 20 / 24 / 32` 的递进尺度。

桌面页面默认：

- 页面水平边距：24
- 区块间距：20~24
- 卡片内边距：16~20
- 小型行间距：8~12

### 6.3 圆角

Aurora Layer 采用“柔和圆角但不过度卡通”：

- XS：8
- SM：12
- MD：16
- LG：20
- XL：24
- Full：999

建议：

- 应用卡片：16~20
- 大面板 / 对话框 / 工作面板：20~24
- 输入框 / 胶囊 / 按钮：999 或 16

### 6.4 阴影

阴影遵循“桌面悬浮感”，不做电商网页重投影：

- 普通卡片：极轻阴影或纯边框
- 浮层 / 通知中心 / 下载管理：中等阴影
- 大容器：靠表面层次和边框，不依赖重阴影

### 6.5 动效

统一节奏：

- Hover：120~160ms
- 切换 / 展开 / 面板进入：180~220ms
- 页面级进场：220~280ms

规则：

- 优先 opacity / transform
- 避免大幅弹跳和移动端式回弹
- 保持桌面理性节奏

## 7. 页面模板规范

### 7.1 推荐页

推荐页承担品牌与发现能力，结构建议为：

- 主 Banner
- 快捷分类 / 专题胶囊
- 精选应用区
- 榜单 / 热门排行
- 主题推荐 / 开发者精选 / 社区生态位

规则：

- Banner 只有一个主视觉区，不能与多个大广告区并列抢注意力。
- 区块标题统一采用“标题 + 查看全部”结构。
- 右侧辅助区可展示今日推荐、专题推荐、运营摘要。
- 推荐页保留展示感，但整体节奏必须规整，不做信息瀑布流。

### 7.2 全部应用 / 搜索结果 / 排行页

三者属于“浏览型列表模板”。

共同规则：

- 顶部展示页标题、结果说明和必要操作
- 分类 / 排序 / 标签使用统一胶囊栏
- 列表卡片网格节奏统一
- footer / 加载更多 / 没有更多使用统一分页组件

差异：

- 全部应用：分类浏览优先
- 搜索结果：结果说明与筛选优先
- 排行页：排名数字、趋势、下载量信息强化

### 7.3 应用详情页

详情页采用“主信息区 + 侧边摘要区”的双栏模板。

左侧主信息区：

- 应用头部
- 主操作按钮
- 截图 / 媒体区
- 介绍
- 应用信息
- 评论区

右侧摘要区：

- 评分与下载量
- 当前版本 / 大小 / 更新信息
- 版本历史摘要
- 开发者其他应用

规则：

- 头部主按钮与次级动作同行展示
- 版本历史强调“当前版本”和“已安装状态”
- 评论区保留评论提交、版本关联和帮助信息
- 页面不做冗余大标题重复

### 7.4 更新页

更新页属于任务型列表页。

结构：

- 头部更新概览
- 批量更新操作区
- 可更新应用列表
- 行级进度 / 状态 / 操作区

规则：

- 视觉上更像待处理工作台，而不是普通应用陈列页
- 批量操作必须醒目但克制
- 任务状态优先级高于应用介绍信息

### 7.5 下载管理

下载管理改为轻工作面板语言。

结构：

- 顶部概览：下载中 / 排队中 / 已完成
- 当前任务
- 等待队列
- 历史记录

规则：

- 使用统一任务项设计
- 行级状态、速度、百分比、错误信息位置固定
- 关闭、清空历史、取消任务等操作遵循危险等级

### 7.6 设置页

设置页不参考当前 Material 风格作为主视觉标准，应并入 Aurora Layer 统一面板语言。

模块建议：

- 语言
- 外观（主题模式 + 主题色）
- 缓存
- 商店偏好
- 关于

规则：

- 采用清晰分组面板
- 主题色板在设置页中需要有真实预览
- 系统设置保持简洁，不引入首页式装饰

## 8. 组件系统

### 8.1 基础组件层

负责全站一致性，不直接承载业务判断。

建议收敛为：

- 按钮系统
- 输入系统
- 选择胶囊 / 标签 / Segment
- 面板 / 弹层 / 对话框
- 通知条 / 通知中心
- 空态 / 错误态 / 骨架屏
- 分页 footer

### 8.2 业务组件层

直接承载商店业务。

包括：

- 应用卡片
- 排行项
- 推荐位卡片
- 详情页摘要卡片
- 评论项
- 版本历史项
- 下载任务项
- 更新项

### 8.3 应用卡片状态规范

继续沿用既有业务规则：

- 未安装：安装
- 已安装无更新：打开
- 已安装可更新：更新
- 安装中 / 更新中：展示进行态
- 排队中：展示等待态

视觉规则：

- 主按钮、次按钮、状态按钮应统一尺寸和重心
- 安装进度与排队状态要有桌面工具感，不得像临时补丁
- Hover 与 active 不影响按钮语义判断

### 8.4 通知与反馈

通知统一走内容区右上角通知中心语义。

类型：

- 成功
- 信息
- 警告
- 错误

规则：

- 通知文案短而明确
- 支持行级操作结果提示
- 禁止继续扩散零散 Snackbar 风格

## 9. 状态表现规范

以下状态必须在 Demo 与未来 Flutter 实现中明确覆盖：

- 空数据
- 首次加载
- 分页加载
- 网络错误
- 无搜索结果
- 未安装
- 已安装
- 可更新
- 安装中
- 排队中
- 失败
- 危险操作确认

每类状态需要同时定义：

- 图形表现
- 文案层级
- 颜色语义
- 按钮动作

---

## 10. 无障碍与桌面交互约束

必须延续现有项目无障碍约定：

- 所有交互控件具备明确语义
- 装饰性图标不参与语义朗读
- 焦点遍历稳定
- 重要按钮热区不小于 48px
- 主题切换后对比度满足可读性要求

桌面交互补充：

- Tooltip 继续保持统一延迟
- 右键菜单保持系统工具风格
- 悬停反馈是重要信息层
- 键盘操作与鼠标操作都必须自然

本规范优先复用现有入口与组件边界：

- `AppShell`：重做壳层样式与布局层级

- `RecommendPage`：增强展示秩序与区块模板
- `AppDetailPage`：改为主副双栏结构
- `DownloadManagerDialog`：重做为轻工作面板语言
- `UpdateAppPage`：改为任务型工作区
