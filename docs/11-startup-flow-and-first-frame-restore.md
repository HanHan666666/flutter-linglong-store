# 启动流程与首帧状态恢复设计

> 文档版本: 1.0 | 创建日期: 2026-03-18  
> 适用范围：Flutter 版玲珑应用商店 Linux 桌面端启动链路、首帧主题/语言恢复、启动关键路径约束。

---

## 一、背景

此前启动链路存在两层“初始化界面”：

1. 路由外的 `AppInitializer`
   - 位于 `MaterialApp.router` 之外
   - 负责初始化 `global / setting / installQueue / launchSequence`
   - 未完成前展示简单的 `CircularProgressIndicator + 正在初始化...`

2. 路由内的 `LaunchPage`
   - 负责环境检测、已安装应用加载、更新检查、安装队列恢复
   - 展示 Logo、步骤文案、进度条、错误态与环境弹窗

这会带来 4 个问题：

- 用户视角存在两个“启动页”，语义重复
- UI 设计文档只定义了一个正式 `LaunchPage`，实现与文档不一致
- 首帧主题、语言依赖外层初始化，架构层次不清晰
- 一些非关键路径工作被错误地塞进启动前，拖慢真实启动路径

因此启动链路需要收敛为：

- **路由外不再存在额外初始化占位页**
- **只保留一个正式 `LaunchPage / LaunchSequence`**
- **首帧主题、语言和基础设置在 Provider `build()` 阶段同步恢复**

---

## 二、目标与非目标

### 2.1 目标

- 启动链路对用户只呈现一个正式启动页
- `MaterialApp` 首帧即使用用户上次保存的语言和主题
- 启动关键路径只保留真正会影响首页正确性的业务步骤
- 非关键路径工作延后，避免污染启动性能
- 让文档、架构、实现三者保持一致

### 2.2 非目标

- 不在本次设计中改动启动页视觉规范
- 不在本次设计中重写 `LaunchSequence` 业务步骤顺序
- 不在本次设计中扩大到所有设置项的统一状态重构
- 不在本次设计中处理设置页已有的 `RadioListTile` 废弃 API 提示

---

## 三、收敛后的启动分层

启动链路收敛后，职责按以下边界划分。

### 3.1 `main()` 负责“系统级引导”

`main()` 只做必须在 `runApp()` 之前完成的工作：

1. `WidgetsFlutterBinding.ensureInitialized()`
2. 日志初始化
3. 单实例检测
4. 窗口初始化
5. `SharedPreferences` 初始化与实例获取
6. `ApiClient` 初始化
7. Hive 初始化
8. 向 `ProviderScope` 注入 `SharedPreferences`
9. `runApp()`

这里的原则是：

- **只能放系统级依赖**
- **不能放业务启动流程**
- **不能再显示一个路由外“正在初始化”页面**

### 3.2 Provider `build()` 负责“首帧状态同步恢复”

以下状态必须在首帧构建时就拿到：

- 当前语言
- 当前主题模式
- 基础设置项，如仓库名
- 用户偏好，如自动检查更新
- 安装队列持久化状态

恢复方式统一为：

- 通过 `sharedPreferencesProvider` 注入 `SharedPreferences`
- 在 Provider `build()` 阶段同步读取本地持久化值
- 直接返回恢复后的初始 state

这样做的原因：

- `MaterialApp.router` 首次 build 就会读取 `locale` 与 `themeMode`
- 如果还要依赖异步 `init()` 再改 Provider 状态，就会先显示默认主题/语言，再闪烁切换
- 启动页属于首帧的一部分，首帧错误就会直接造成视觉跳变

### 3.3 `LaunchPage / LaunchSequence` 负责“正式业务启动流程”

正式启动流程统一放到 `LaunchSequence`：

1. 环境检测
2. 已安装应用初始化
3. 更新检查
4. 安装队列恢复
5. 完成后进入首页

`LaunchPage` 只负责：

- 展示正式启动 UI
- 监听 `LaunchSequence` 状态
- 显示错误态与环境弹窗
- 完成后导航到首页

## 四、首帧一致性的业务细节

### 4.1 为什么语言和主题必须同步恢复

`MaterialApp.router` 在首次 build 时就依赖以下值：

- `locale`
- `themeMode`
- 基于主题计算的 `NativeMenuThemeSync`

如果这些值在应用启动后再异步写入：

- 启动页会先以默认中文或系统主题渲染
- 随后再切到用户保存的语言/主题
- 会出现明显闪烁
- 与“启动页是品牌页、第一印象必须稳定”的目标冲突

因此这里的规则是：

- **语言、主题、基础设置属于首帧依赖**
- **首帧依赖必须同步恢复**
- **禁止再用外层占位页掩盖这类恢复逻辑**

### 4.2 为什么安装队列也允许在 Provider `build()` 恢复

安装队列的本地状态恢复有两个特性：

- 数据源是本地 `SharedPreferences`
- 恢复过程不依赖网络，也不依赖路由树稳定后再写入状态

因此它适合像主题/语言一样在 Provider `build()` 里同步恢复“本地快照”。  
但要注意：

- 这里恢复的是**本地持久化状态**
- 真正的“崩溃恢复校验”和“结合已安装列表纠偏”仍属于 `LaunchSequence` 的 `queueRecovery`

也就是说：

- **build 恢复本地快照**
- **launch 再做业务级校正**

## 五、启动关键路径与延后策略

### 5.1 必须阻塞首页进入的步骤

以下步骤属于启动关键路径，必须在首页交互前完成：

- 环境检测
- 已安装应用列表初始化
- 更新检查
- 安装队列恢复与纠偏

原因如下：

- 环境未就绪时，首页继续交互没有意义
- 已安装列表是卡片状态、我的应用、更新页的基础输入
- 更新检查影响红点、更新页和卡片“更新/打开”状态
- 安装队列恢复影响下载管理、按钮 loading/progress 和异常恢复

### 5.2 必须延后的步骤

以下工作不得继续占用启动关键路径：

- 设置页缓存大小统计
- 与设置页展示直接相关但不影响首页正确性的派生信息
- 可在进入特定页面后再计算的辅助数据

典型例子是 `settingProvider` 的缓存大小：

- 它需要扫描 Hive/临时目录
- 这个过程与首页首帧正确性无关
- 放在启动前只会延长正式启动时间

因此这里的规则是：

- **不影响启动页正确展示和首页业务判定的逻辑，一律延后**
- **进入设置页后再异步刷新缓存体积**

## 六、正式启动流程时序

收敛后的冷启动主路径如下：

1. `main()` 完成系统级引导并注入 `SharedPreferences`
2. `ProviderScope` 创建各 Provider
3. `globalAppProvider / settingProvider / installQueueProvider` 在 `build()` 同步恢复本地状态
4. `MaterialApp.router` 首帧即使用正确语言和主题
5. 路由初始进入 `LaunchPage`
6. `LaunchPage` 在首帧后触发 `LaunchSequence.runSequence()`
7. `LaunchSequence` 依次执行环境检测、已安装应用初始化、更新检查、安装队列恢复
8. 如环境异常，交由 `LinglongEnvDialog` 处理
9. 完成后跳转首页

这里要强调两个边界：

- `main()` 不做业务启动
- `LaunchSequence` 不负责首帧主题/语言恢复

---

## 七、错误处理与恢复原则

### 7.1 路由外不再承载错误展示

删除 `AppInitializer` 后，不再允许在 `MaterialApp.router` 之外单独渲染“初始化失败页”。  
启动错误统一由正式 `LaunchPage` 展示。

好处：

- 用户看到的错误入口只有一个
- 启动流程的 loading / error / retry / skip 都集中在一个状态机里
- 视觉和交互不会在两套页面之间漂移

### 7.2 恢复原则

- 环境检测失败：由环境弹窗接管，允许重新检测、安装环境、跳过或退出
- 业务步骤失败：由 `LaunchPage` 错误区统一承载重试/跳过
- 非关键路径工作失败：不阻塞首页，只记录日志并允许后续页面独立刷新

---

## 八、实现约束

后续开发必须遵守以下约束：

1. 禁止新增第二个路由外启动占位页
2. 禁止把业务启动步骤重新放回 `main()`
3. 禁止把首帧依赖改回异步 `init()` 后再写状态
4. 禁止把设置页缓存大小之类的非关键路径逻辑重新塞回启动前
5. 启动页之外的页面，不得自行补做启动阶段全局初始化

---

## 九、验证要求

本设计落地后至少需要验证：

### 9.1 功能验证

- 冷启动只出现一个正式启动页
- 启动页首次显示即为用户上次保存的语言
- 启动页首次显示即为用户上次保存的主题
- 环境异常时仍能正常弹出环境处理对话框
- 启动完成后能正确进入首页

### 9.2 回归验证

- 设置页切换语言后，重启应用首帧语言正确
- 设置页切换主题后，重启应用首帧主题正确
- 安装队列持久化后，重启仍能恢复本地状态
- 设置页缓存大小仍能在进入设置页后正常刷新

### 9.3 自动化验证

- 为首帧恢复增加 Provider 级单元测试
- 启动 smoke 测试需覆盖“首帧主题/语言正确”
