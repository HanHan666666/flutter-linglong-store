# 右上角通知中心设计

## 背景

Flutter 商店当前的大部分操作反馈仍依赖 `ScaffoldMessenger.of(context).showSnackBar(...)`。这带来几个直接问题：

1. 视觉表现仍是偏底部的 Material SnackBar，和当前桌面端应用外壳不协调。
2. 通知调用入口分散在页面、弹窗和服务层，缺少统一样式与统一交互约束。
3. 现有 `SnackBarThemeData` 只能调整颜色、圆角和 floating 行为，无法自然实现“内容区右上角堆叠弹出”。
4. 一些服务层逻辑（例如卸载服务）也直接依赖 `BuildContext + ScaffoldMessenger`，这让通知能力难以复用、难以测试。

用户已确认的目标是：

1. 通知改为内容区右上角弹出，而不是底部样式。
2. 避开自定义标题栏和窗口控制按钮，不遮挡核心交互。
3. 多条通知时最多堆叠 `3` 条。
4. 采用“操作型通知”风格，至少支持关闭按钮，并为后续动作按钮预留能力。
5. 改造范围为全局统一替换，而不是单页试点。

## 目标

1. 建立统一的全局通知中心，替代当前分散的 `showSnackBar` 调用。
2. 在不引入额外第三方通知库的前提下，实现桌面端更协调的右上角通知卡片。
3. 保持通知逻辑与现有分层架构一致：UI 负责展示，Application 层负责调度，服务层不再直接绑死 `ScaffoldMessenger`。
4. 为 success / error / info / warning 四类通知建立统一视觉与交互约束。
5. 为后续需要携带操作按钮的通知预留统一 API，而不是继续让各页面各写一套弹出逻辑。

## 非目标

1. 不在本次改造中引入系统级桌面通知。
2. 不改动业务文案来源与国际化机制，仅替换展示通道。
3. 不重构所有页面的业务流程，只替换用户反馈出口。
4. 不引入第三方 toast/snackbar 框架，避免桌面端样式和行为受外部包限制。

## 现状分析

### 1. 通知入口分散

当前仓库内存在多处直接调用 `ScaffoldMessenger.of(context).showSnackBar(...)` 的逻辑，典型包括：

- 应用卸载服务
- 应用详情页评论提交
- 设置页多个反馈入口
- 应用卡片启动应用结果提示
- 环境检测、反馈弹窗、玲珑进程等桌面特有面板

这意味着“仅改 theme”无法完成目标，必须把通知出口收口。

### 2. 视口挂载点必须是 Shell 感知的

应用入口 `MaterialApp.router.builder` 虽然存在统一包装层，但真正定义“标题栏 + 侧边栏 + 右侧内容区”的布局是在 `ShellRoute -> AppShell` 内完成的。因此通知视口不能只挂在 `MaterialApp.router.builder` 里直接按窗口坐标定位，否则会天然对齐整个窗口右上角，而不是右侧内容区右上角。

本次通知中心的正式挂载点应放在 `AppShell` 内部的 `Stack`，并覆盖在右侧内容容器之上。这样可以：

1. 精确避开标题栏和窗口控制按钮。
2. 让通知始终对齐主内容区，而不是窗口边框。
3. 避免 LaunchPage 这类 Shell 外页面被错误纳入通知定位体系。

### 3. 标题栏与窗口控制区域需要避让

当前自定义标题栏高度固定，右上角窗口控制按钮贴齐窗口右侧。如果通知直接贴窗口右上角，容易遮挡最小化/最大化/关闭按钮。因此通知位置必须锚定在“标题栏下方的内容区右上角”，而不是整个窗口的右上角。

## 方案比较

### 方案 A：继续使用 SnackBar，仅调主题和 margin

做法：

1. 保留所有 `showSnackBar` 调用。
2. 通过 `SnackBarThemeData`、`margin`、`width` 等参数让视觉看起来接近右上角卡片。

优点：

1. 改动最小。
2. 不需要新增基础设施。

缺点：

1. `SnackBar` 语义仍然偏底部反馈，不适合右上角堆叠式桌面通知。
2. 多条通知堆叠、统一关闭按钮、动作按钮、窗口级定位都不自然。
3. 仍然保留大量分散调用，后续维护成本高。

结论：不采用。

### 方案 B：引入第三方 toast/notification 包

做法：

1. 新增第三方包，如 toast/snackbar/overlay notification 框架。
2. 用第三方提供的 overlay 管理器统一替换现有 `SnackBar`。

优点：

1. 起步速度可能更快。
2. 部分动画和队列管理可直接复用。

缺点：

1. 桌面端细节、主题一致性、层级控制会受包能力限制。
2. 当前项目对 UI 一致性和长期维护要求高，引入第三方后可控性下降。
3. 后续如果需要和现有标题栏、对话框层级、路由壳体深度耦合，二次适配成本更高。

结论：不采用。

### 方案 C：自建全局 Overlay 通知中心（推荐）

做法：

1. 新增统一通知数据模型与调度控制器。
2. 在应用根部挂一个通知视口组件，统一渲染右上角通知栈。
3. 各页面和服务统一走通知中心 API，而不是直接调 `ScaffoldMessenger`。

优点：

1. 位置、动画、堆叠、动作按钮、主题都可精确控制。
2. 能彻底解决通知调用分散的问题。
3. 与当前项目的分层架构最匹配，后续可持续演进。

缺点：

1. 需要一次性建立通知基础设施。
2. 需要逐步迁移旧调用点。

结论：采用方案 C。

## 选型

采用“全局 Overlay 通知中心”方案。

原因：

1. 这是唯一能同时满足位置、视觉、交互和统一入口要求的方案。
2. 当前代码结构已经具备合适的全局挂载点，技术可行性高。
3. 不依赖外部包，更符合当前仓库对高性能、高一致性、可维护性的要求。

## 架构设计

### 1. 数据层：通知实体

新增轻量通知实体，例如：

- `id`
- `message`
- `type`：success / error / info / warning
- `duration`
- `actionLabel`
- `actionId`
- `dismissible`

其中可展示状态对象只保存 `actionId`，不直接把闭包放进状态树。具体动作处理器由控制器内部 registry 维护，这样可以避免：

1. 把不可序列化闭包塞进全局状态。
2. 让比较、调试和测试行为变脆弱。
3. 通知提前移除后仍残留无主回调。

### 2. 应用层：通知控制器

新增 `AppNotificationController`，职责包括：

1. 接收新增通知请求。
2. 维护通知列表顺序。
3. 限制同时显示最多 `3` 条。
4. 管理自动消失计时器。
5. 维护 action handler registry。
6. 支持手动关闭与动作点击后的清理。

该控制器只负责状态和生命周期，不负责具体渲染。

控制器生命周期契约：

1. 该 provider 是 app-wide 单例，必须保持 keepAlive。
2. timer 与 action handler 在通知移除时必须同步清理。
3. 应用退出前不要求持久化通知队列。

### 3. 展示层：通知视口与通知卡片

新增两个展示组件：

1. `AppNotificationViewport`
   - 挂在 `AppShell` 的 `Stack` 中
   - 负责把通知栈固定在内容区右上角
   - 负责与标题栏高度、内容区内边距对齐

2. `AppNotificationCard`
   - 负责单条通知的视觉、动作按钮、关闭按钮、进出场动画
   - success / error / info / warning 使用同一布局，不同语义色

### 4. 页面与服务的调用方式

通知调用统一改为 helper / provider 形式，例如：

- `showSuccess(...)`
- `showError(...)`
- `showInfo(...)`
- `showWarning(...)`

页面层可以直接通过 `ref.read(...)` 或统一 helper 调用；服务层本次统一改为返回 typed result，由调用侧在展示边界发通知，不再允许服务内部直接持有通知实现。

层间边界约束：

1. `application/services/*` 不允许 import `presentation/notifications/*`。
2. 应用层服务不允许持有 `BuildContext`、`WidgetRef` 或展示层 helper 跨 `await` 使用。
3. 对于像卸载服务这样的链路，正式契约固定为“服务返回 typed result，由调用侧统一发通知”；本次不采用“在 provider 边界注入回调让服务内部发通知”的变体。
4. 跨层共享的通知实体与 dispatcher 协议只允许放在 `application/notifications/*`，不允许由 `application/*` 反向依赖 `presentation/*`。

## 交互与视觉约束

### 1. 位置

1. 通知显示在内容区右上角。
2. 顶部避开自定义标题栏。
3. 右侧预留与窗口控制区及页面边缘一致的安全间距。

### 2. 堆叠规则

1. 同时最多显示 `3` 条。
2. 新通知从上方列表顶部插入，旧通知向下顺移。
3. 超过上限时，最旧通知优先移除。

### 3. 交互

1. 默认自动消失。
2. 悬停时暂停自动消失计时，避免用户来不及点操作。
3. 右上角始终提供关闭按钮。
4. 如果传入 `actionLabel + actionId`，在卡片右侧显示次级操作按钮。
5. 动作按钮契约固定为：点击后先执行 handler，再关闭当前通知；本次不支持“动作完成后按结果决定是否保留通知”的分支行为。

### 4. 动画

1. 进入：右侧滑入 + 淡入。
2. 退出：淡出 + 轻微右移。
3. 堆叠重排时使用平滑位移动画，避免突兀跳动。

## 迁移策略

### 阶段 1：建立新基础设施

1. 新增通知模型、控制器、视口、卡片组件。
2. 在 `AppShell` 右侧内容区上层接入视口。
3. 完成单元和 Widget 测试。

### 阶段 2：迁移高频入口

优先替换这些用户感知最强的通知：

1. 应用启动结果反馈。
2. 卸载成功/失败反馈。
3. 应用详情评论提交反馈。
4. 设置页操作反馈。
5. 反馈弹窗、环境检测、进程管理面板。

### 阶段 3：清理旧 SnackBar 入口

1. 全局搜索剩余 `showSnackBar` 调用。
2. 用统一通知 helper 替换。
3. 最终验收口径应为业务代码内零直接 `ScaffoldMessenger/showSnackBar` 调用，本次不预留 allowlist。
4. 清理已经不再有业务意义的 `SnackBarThemeData` 定制依赖。

## 风险与应对

### 1. 服务层仍依赖 BuildContext

风险：服务层如果继续直接调用 `ScaffoldMessenger`，通知中心无法彻底统一。

应对：

1. 服务层统一改为返回 typed result，由 provider/page 边界决定是否展示通知。
2. 本次先处理已有直接 `showSnackBar` 的服务层入口，避免新旧体系并存太久。

### 2. 层级冲突

风险：通知可能与对话框、截图预览、菜单等 overlay 冲突。

应对：

1. 视口固定挂载在 `AppShell` 的 `Stack` 中，并位于右侧内容容器之上。
2. 必要时通过 `IgnorePointer` 与局部命中策略限制通知栈只拦截卡片区域。
3. 如果后续需要覆盖 Shell 外页面，必须单独设计第二套定位策略，而不是复用本次内容区锚点。

### 3. Shell 外页面范围

风险：LaunchPage 当前位于 Shell 外，不具备和主内容区一致的定位基准。

应对：

1. 本次通知中心默认覆盖 AppShell 内页面。
2. LaunchPage 暂不纳入首批替换范围，除非后续确认其确实需要同类通知。

### 3. 桌面窗口尺寸变化

风险：窗口缩放、最大化时，通知偏移可能不协调。

应对：

1. 使用相对布局而非写死屏幕坐标。
2. 偏移依赖标题栏高度和统一 spacing 常量。

## 测试策略

1. 单元测试：
   - 控制器入队 / 出队
   - 最多保留 3 条
   - 自动消失
   - 手动关闭

2. Widget 测试：
   - 通知视口右上角渲染
   - 多条堆叠顺序
   - 动作按钮点击
   - 悬停暂停自动消失
   - 对话框场景下的层级与命中行为

3. 回归验证：
   - 卸载反馈
   - 启动应用反馈
   - 设置页反馈
   - 评论提交反馈
   - 对话框提交反馈
   - 进程/菜单驱动反馈
   - 深色 / 浅色主题切换

## 验证标准

1. 全局统一替换目标范围内的旧 `SnackBar` 调用。
2. 通知稳定显示在内容区右上角，不遮挡标题栏和窗口控制按钮。
3. 连续触发 3 条以上通知时，展示和淘汰规则符合预期。
4. 动作按钮、关闭按钮、自动消失在桌面端交互自然。
5. 业务代码内零直接 `ScaffoldMessenger/showSnackBar` 调用。
6. `flutter analyze` 无新增错误与警告，新增测试通过。
