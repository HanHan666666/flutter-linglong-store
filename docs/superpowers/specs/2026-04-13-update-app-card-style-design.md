# Update App 页面卡片风格统一设计

## 1. 背景

`lib/presentation/pages/update_app/update_app_page.dart` 当前列表项采用单独手写的 `Card + BorderSide(outlineVariant)` 方案，卡片边框感偏重，和项目中其他主列表页复用的 `AppCard` 视觉风格存在明显差异。当前差异主要表现为：

- 更新页列表项呈现较强的灰色描边框感；
- 其他主列表页更偏向 surface 背景卡片、弱边界和轻量 hover 反馈；
- 造成更新页在整体观感上与首页、排行、全部应用、我的应用等页面不够统一。

本次需求仅要求统一列表 item 的卡片风格，不允许改动页面整体布局和业务交互。

## 2. 目标

在不改变 `update_app` 页面布局结构的前提下，使更新列表 item 的卡片风格与项目现有主流列表卡片观感保持一致。

### 保持不变

- 页面头部区域不调整；
- 列表结构、间距层级和信息排布不调整；
- item 内部仍保持：左侧图标、中间名称与版本信息、右侧更新按钮、下方进度消息/更新说明；
- 更新、取消、进度、网络速度等业务逻辑不调整；
- 不新增新的业务状态或交互入口。

### 本次要解决的问题

- 弱化当前 item 的“灰框框”感；
- 使卡片背景、圆角、边界表现更接近全局卡片样式；
- 在桌面端提供与其他页面一致的轻量 hover 反馈。

## 3. 方案对比

### 方案 A：保留现有结构，只统一卡片外观（采用）

做法：继续保留 `_UpdatableAppItem` 的现有内容结构与按钮逻辑，仅替换外层容器样式，使其对齐 `lib/presentation/widgets/app_card.dart` 的视觉语言。

优点：

- 改动面最小；
- 风险最低；
- 最符合“只改卡片风格，不改其他 UI 布局”的需求边界；
- 不需要为更新页专门扩展 `AppCard` 能力。

缺点：

- 更新页仍保留独立 item 实现，样式统一依赖人工对齐而非完全复用组件。

### 方案 B：直接复用 `AppCard`

做法：将更新页列表项改造成 `AppCard`，再补充版本信息、进度消息与更新说明区域。

优点：

- 组件复用程度最高；
- 长期一致性更强。

缺点：

- `AppCard` 当前信息结构偏“名称 + 描述”，与更新页“版本对比 + 状态文案”的结构不完全一致；
- 需要扩展 `AppCard`，改动边界容易超出本次需求；
- 风险高于仅调整外观。

### 方案 C：仅弱化边框，不做完整风格对齐

做法：保留现有 `Card`，只微调 border、颜色和圆角。

优点：

- 改动最保守。

缺点：

- 只能缓解突兀感，无法真正实现与其他页面风格统一。

## 4. 最终方案

采用方案 A：保留当前 `_UpdatableAppItem` 的结构和交互逻辑，只调整卡片视觉表现。

### 具体设计

1. **保留现有布局结构**
   - 不调整 `Row + 下方补充文案` 的层级；
   - 不改应用图标、名称、版本信息、更新按钮、进度文案、更新说明的布局顺序。

2. **统一卡片背景与圆角表达**
   - 将当前强调描边的 `Card` 外观改为更接近 `AppCard` 的 surface 背景卡片；
   - 圆角、内边距保持与现有项目卡片风格一致。

3. **去掉强灰色描边感**
   - 不再使用当前这种显式、存在感较强的 `outlineVariant` 外框表现作为主视觉；
   - 如保留边界，也仅允许使用非常弱的边界辅助层次，而不能让列表项重新呈现“灰框框”观感。

4. **补充轻量桌面 hover 反馈**
   - 参考 `AppCard` 的处理方式，为 item 增加轻量 hover 阴影或等效反馈；
   - 反馈必须克制，不能改变布局尺寸，也不能造成视觉抖动。

5. **不改按钮与状态逻辑**
   - `InstallButton` 的状态流转、下载速度展示、取消行为、禁用规则均保持现状；
   - 仅允许围绕容器视觉进行改动。

## 5. 影响范围

预计仅修改以下文件：

- `lib/presentation/pages/update_app/update_app_page.dart`

不扩展到公共组件，不修改其他页面，不新增新的主题配置项。

## 6. 验收标准

完成后应满足：

1. 更新页列表 item 不再呈现突兀的灰色描边框感；
2. 卡片观感与其他主列表页更一致；
3. 页面头部、列表布局、item 信息排布保持不变；
4. 更新、取消、进度、网络速度展示逻辑无回归；
5. 仅通过样式层调整达成目标，不引入额外业务复杂度。

## 7. 非目标

以下内容不在本次范围内：

- 将更新页完全重构为复用 `AppCard`；
- 调整页面头部布局；
- 修改按钮文案、交互规则或业务状态机；
- 修改列表数据加载、刷新、排队或安装流程。
