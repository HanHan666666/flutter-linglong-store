# 标题栏搜索候选设计

**Date:** 2026-05-27

## 背景

当前桌面端搜索入口统一收敛在标题栏搜索框：

1. 用户在 `lib/presentation/widgets/title_bar.dart` 输入后，按 `Enter` 或点击搜索图标，会通过 `context.goToSearch()` 进入 `lib/presentation/pages/search_list/search_list_page.dart`。
2. 搜索结果页由 `searchProvider` 驱动，使用 `/visit/getSearchAppList` 拉取分页应用列表。
3. 应用详情页支持沿用列表入口传入的 `appId + arch + repoName + module`，但当前 `RecommendAppInfo -> InstalledApp` 的映射只透传了 `arch`，没有把 `repoName` 与 `module` 一并带过去。

本次需求是在标题栏搜索框中增加“搜索候选”能力，且候选词来源直接复用现有搜索接口，不新增接口或新业务语义。

## 目标

1. 用户在标题栏输入关键词时，显示远端搜索候选。
2. 候选项只展示应用名称，不显示图标、副标题、评分或按钮。
3. 用户点击候选项后，直接进入该应用详情页。
4. 用户按 `Enter` 时保持现状，继续进入搜索结果页，不抢占为“打开第一条候选”。
5. 保持标题栏现有视觉结构和搜索入口不变，不引入第二个搜索框。

## 非目标

1. 不新增独立搜索建议接口。
2. 不修改搜索结果页的分页、排序或卡片布局。
3. 不在候选面板中增加图标、描述、快捷安装等复杂交互。
4. 不改造为键盘上下键选中候选；本次只要求鼠标点击候选跳详情。
5. 不在页面其他搜索输入框（如“我的应用”页本地搜索）同步接入远端候选。

## 现状分析

### 1. 标题栏搜索框目前只有提交，没有候选层

`lib/presentation/widgets/title_bar.dart` 内的 `_TitleSearchBox` 当前只维护：

1. `TextEditingController`
2. `FocusNode`
3. 聚焦态边框
4. 清空与提交搜索

没有候选状态、远端请求、下拉面板或点击跳详情逻辑。

### 2. 现有 `searchProvider` 不适合直接承载候选

`lib/application/providers/search_provider.dart` 的职责是完整搜索结果页状态：

1. 当前搜索词
2. 结果列表
3. 首屏 loading
4. 分页加载更多
5. 搜索页刷新

如果标题栏候选直接复用该 provider，会带来两个问题：

1. 用户在首页输入候选时，会污染真正的搜索结果页状态。
2. 候选请求只需要一小页轻量数据，不应和搜索页的分页/刷新耦合。

因此候选应使用单独的轻量状态层。

### 3. 搜索接口已具备候选所需字段

后端 `/visit/getSearchAppList` 当前返回的列表项已包含：

1. `appId`
2. `zhName/appName`
3. `arch`
4. `module`
5. `repoName`

且后端 SQL 已按 `app_id` 做分组，优先返回单个应用的最新版本主记录，因此候选面板可以直接把返回结果当成“应用候选”，不需要前端再做二次去重聚合。

### 4. 详情身份透传存在补齐空间

仓库现有约定要求“列表页进入详情页时必须原样透传 `appId + arch + repoName + module`”。当前搜索结果页虽然会把 `RecommendAppInfo` 转成 `InstalledApp` 再跳详情，但 `RecommendAppInfo` 尚未承载 `repoName` 与 `module`，这会导致：

1. 搜索结果进入详情时丢失部分身份字段。
2. 新增标题栏候选点击详情时，如果继续沿用现有映射，也会丢失同样的信息。

本次需要把这层补齐，避免详情页命中错误条目或走回退匹配。

## 方案比较

### 方案 A：在标题栏内直接增加候选下拉（推荐）

做法：

1. 在 `_TitleSearchBox` 内增加防抖输入监听。
2. 通过独立候选 provider 请求 `/visit/getSearchAppList` 的第一页小页量数据。
3. 在搜索框下方以内联 overlay/popup 的形式展示候选列表。
4. 点击候选时直接跳详情；按 `Enter` 保持现有跳搜索页行为。

优点：

1. 改动集中在标题栏链路，最符合当前需求。
2. 不污染搜索结果页状态。
3. 可以直接复用现有搜索接口和详情页路由能力。

缺点：

1. 需要补一层轻量候选状态。
2. 要处理焦点丢失、点击候选与关闭面板的细节。

### 方案 B：复用 `searchProvider`，根据场景区分“候选态”和“结果页态”

做法：

1. 在 `searchProvider` 中新增字段区分 `suggestion/results` 场景。
2. 标题栏和搜索页共用一个 provider。

优点：

1. 表面上 provider 数量更少。

缺点：

1. 标题栏输入会干扰搜索结果页的状态。
2. 状态机会明显复杂化。
3. 后续维护时更难判断哪个字段属于哪个入口。

结论：不采用。

### 方案 C：重构为通用 SearchBar 组件并统一接入多个页面

做法：

1. 以 `lib/presentation/widgets/search_bar.dart` 为基础重做远端候选。
2. 标题栏、搜索页、我的应用页都逐步统一。

优点：

1. 长远复用性更高。

缺点：

1. 本次需求只是标题栏候选，直接升级为通用组件会扩大改动面。
2. 现有 `search_bar.dart` 是本地字符串过滤模型，不适合作为这次最小变更的基础。

结论：本次不采用。

## 选型

采用方案 A：在标题栏搜索框内直接增加远端候选下拉，并引入独立轻量候选 provider。

## 设计细节

### 1. 候选状态单独建模

新增独立的标题栏搜索候选 provider，仅负责：

1. 当前输入对应的候选请求状态
2. 候选项列表
3. 是否正在加载
4. 错误后的静默回收

该 provider 不负责：

1. 搜索结果页分页
2. 搜索结果页总数
3. 搜索页刷新
4. 全局缓存

设计原则是“标题栏只拿轻量候选，不和搜索结果页共享状态机”。

### 2. 请求策略

候选请求策略固定为：

1. 输入值 `trim()` 后为空：不请求，立即清空候选。
2. 输入值非空：使用 `300ms` 防抖触发请求。
3. 仅请求第一页。
4. `pageSize` 固定为 `8`，限制网络和渲染成本。
5. 请求仍走 `/visit/getSearchAppList`，复用现有 `arch` 与 `lan` 传递方式。

这样可以兼顾标题栏的输入响应速度和接口负载，不会把完整搜索页的 20 条首屏结果照搬进候选。

### 3. 候选展示规则

候选面板只在以下条件同时满足时显示：

1. 输入框处于 focus 状态。
2. 当前输入非空。
3. 候选列表非空。

候选项展示内容只保留应用名称：

1. 文本取搜索接口返回的 `appName`
2. 单行省略
3. 不显示 icon、description、version、repoName

候选面板关闭时机：

1. 输入清空时关闭
2. 输入框失焦时延迟关闭，给点击候选预留事件窗口
3. 点击候选后立即关闭
4. 提交 `Enter` 搜索后立即关闭

### 4. 点击与回车行为

交互规则固定为：

1. 点击候选项：直接 `goToAppDetail()`，并传入由候选项映射出的 `InstalledApp`
2. 按 `Enter`：继续走 `goToSearch(query)`，不自动打开第一条候选
3. 点击搜索图标：继续走 `goToSearch(query)`

这样标题栏会同时支持：

1. “我知道我要哪个应用，点候选直达详情”
2. “我想看完整结果页，按 `Enter` 搜索”

### 5. 详情页透传字段补齐

为保证候选与搜索结果点击都能稳定命中正确详情条目，需要补齐：

1. `RecommendAppInfo` 新增 `module` 与 `repoName`
2. `mapAppListToRecommendApps()` 映射时带上后端返回的 `module` 与 `repoName`
3. `RecommendAppInfo.toInstalledApp()` 把 `module` 与 `repoName` 一并写入 `InstalledApp`

这样：

1. 搜索结果页进入详情会更符合既有仓库约定
2. 标题栏候选进入详情可复用同一套模型转换
3. 详情页 `loadDetail()` 能优先用精确身份字段请求后端详情接口

### 6. 可访问性与视觉约束

候选列表继续遵循现有无障碍约定：

1. 候选项使用 `Semantics` 标注，朗读应用名
2. 可点击项保持最小可用交互高度
3. 不引入装饰性无意义朗读文本

视觉上保持：

1. 标题栏搜索框本体尺寸和边框逻辑不变
2. 候选面板在搜索框下方展开
3. 面板宽度与搜索框对齐
4. 面板高度有上限，超出时滚动

## 影响范围

### 直接修改

1. `lib/presentation/widgets/title_bar.dart`
2. 新增 `lib/application/providers/title_search_suggestions_provider.dart`
3. `lib/domain/models/recommend_models.dart`
4. `lib/data/mappers/app_list_mapper.dart`
5. 标题栏搜索与候选相关测试

### 间接受益

1. 搜索结果页进入详情的身份透传更完整
2. 后续若其他页面也要复用候选 provider，可以在不影响搜索页的前提下扩展

## 风险与缓解

### 风险 1：频繁输入导致过多接口请求

缓解：

1. 使用 `300ms` 防抖
2. 只请求第一页 `8` 条
3. 空输入立即清空，不发送请求

### 风险 2：候选点击与失焦关闭竞争

缓解：

1. 失焦后延迟关闭
2. 候选点击逻辑先执行跳转，再收起候选层

### 风险 3：详情身份字段仍不完整

缓解：

1. 在搜索列表 DTO -> `RecommendAppInfo` -> `InstalledApp` 全链路补齐 `module + repoName`
2. 通过测试覆盖候选点击和搜索结果点击的透传行为

### 风险 4：标题栏状态过重影响首帧或输入流畅度

缓解：

1. 候选状态单独封装，不在 `build()` 中做同步 IO
2. 面板只消费轻量名称列表与少量详情透传字段
3. 候选请求失败时静默收起，不阻塞用户继续按 `Enter` 搜索

## 测试策略

### 单元测试

新增候选 provider 测试，覆盖：

1. 空输入时清空候选且不请求接口
2. 非空输入时请求第一页 `8` 条
3. 成功时返回候选列表
4. 失败时清空候选并结束 loading

### Widget 测试

补充标题栏搜索测试，覆盖：

1. 输入后显示候选名称
2. 点击候选后进入详情页
3. 按 `Enter` 仍进入 `/search_list?q=...`
4. 清空输入后候选消失

### 回归测试

补充模型映射测试或现有测试断言，覆盖：

1. 搜索结果项映射后保留 `arch + module + repoName`
2. `RecommendAppInfo.toInstalledApp()` 不再丢失 `module + repoName`

## 预期结果

完成后，标题栏搜索将支持两条清晰路径：

1. 输入关键词后直接点击候选，快速进入应用详情页
2. 输入关键词后按 `Enter`，继续进入完整搜索结果页

同时，搜索链路进入详情页时会统一保留 `appId + arch + repoName + module` 身份字段，减少详情误匹配和后端回退查询带来的不确定性。
